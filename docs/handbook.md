# financialanalysis 项目手册

## 1. 项目定位

financialanalysis 是一套面向债券发行主体研究的开源技术仓库，重点解决四类问题：资料采集、文档解析、财务分析、运行治理。

它不是单脚本工具，也不是单纯的报表生成器，而是一条围绕债券主体研究展开的可复用工作流。核心思路是把“拿材料、读材料、做分析、出结果、留痕迹”拆成独立模块，让每一步都可复核、可交接、可继续开发。

## 2. 这个项目做什么

这个项目处理的是债券研究里最常见的一类任务：围绕发行主体的财务与信用信息做结构化分析。

主要看什么：

- 主体是谁，报告属于哪一年、哪一类。
- 财报正文和附注里有哪些关键证据。
- 债务结构、现金覆盖、受限资金、担保、或有事项、条款风险是否有压力。
- 分析结论能否回到原始材料和中间产物。

因此，这个项目的输出不是摘要，而是面向债券研究的分析工作流产物：工作底稿、分析报告、结构化结果和运行记录。

## 3. 业务逻辑

整个项目的业务逻辑可以理解为一条从输入到结论再到留痕的链路。

1. 从官方来源获取财报、公告、评级报告和披露材料。
2. 用文档解析能力把 PDF、Word、PPT、图片等材料转成可读的中间文本。
3. 识别主体、期间、报告类型和附注范围。
4. 优先读取附注，抽取债务、现金、受限资金、担保、或有事项、现金流、资本化利息等证据。
5. 将正文与附注信息合并为可核对的工作底稿。
6. 基于工作底稿生成分析结论、风险判断和结构化输出。
7. 把稳定口径沉淀为知识输入，供后续案例复用和回滚。

这意味着项目不是在做“把财报读成摘要”，而是在做“面向债券研究的主体信用分析”。

## 4. 三个核心模块

### 4.1 `financial-analyzer`

这是项目的财务分析技能主模块。

它负责的不是简单清洗数据，而是围绕债券发行主体做结构化分析：先识别报告类型和主体，再定位附注与核心证据，接着抽取债务、现金、受限资金、担保、或有事项和现金流等信息，最后形成可复核的工作底稿、分析报告和结构化结果。

它的核心约束是：

- 附注优先。
- 先草稿后正式。
- 先复核后采纳。
- 草稿产物不能直接当最终结论。

### 4.2 `chinamoney`

这是数据采集模块，负责从中国货币网等官方来源获取财报、评级报告和披露材料。

它解决的是“怎么稳定拿到材料”的问题。实践上，它更像一个采集入口：先建立会话，再访问接口或下载链接，最后把原始材料交给后续解析和分析环节。

### 4.3 `mineru`

这是文档解析模块，负责把非结构化文档转换成可读中间文本。

它解决的是“怎么把原始文件变成可分析文本”的问题。典型场景包括年报解析、附注定位、扫描件 OCR、图片提取和多格式文档统一转换。

## 5. 工作流

标准工作流如下：

```text
任务定义(issuer-year)
  -> 数据采集层（ChinaMoney）
  -> 文档理解层（MinerU）
  -> 文档标准化层（识别主体、期间、附注范围）
  -> 分析引擎层（financial-analyzer）
  -> 运行治理层（manifest / 状态 / 重跑）
  -> 正式产物层（工作底稿、报告、结构化输出）
  -> 知识治理层（review / adopt / rollback）
```

每一层都只做一件事：

- 采集层负责把材料拿回来。
- 解析层负责把材料读出来。
- 标准化层负责把材料组织成统一结构。
- 分析层负责形成债券主体的财务与信用判断。
- 运行治理层负责记录这次运行发生了什么。
- 正式产物层负责输出可交付结果。
- 知识治理层负责沉淀可以复用的经验和口径。

## 9. 开发原则

- 先读文档，再动手。
- 一次只改一类问题。
- 路径、字段名和产物名尽量稳定。
- 不把草稿产物当成正式结论。
- 不把运行态数据写回 skill 目录。
- 不把采集、解析、分析和导出混成一个不可拆分的脚本。

## 10. 适合继续扩展的方向

- 增强数据采集稳定性。
- 增强 PDF 和扫描件解析能力。
- 完善附注优先的分析口径。
- 补齐数据契约和排障文档。
- 增强正式产物与知识治理的可追溯性。
